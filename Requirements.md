## 1. 프로젝트 개요

### 1.1 목적

부동산 경매 물건에 대해 아래 항목을 자동화하여 **단기 매매사업자의 의사결정을 지원**하는 시스템 구축

- 데이터 수집
- 권리 분석
- 시세 비교
- 수익성 분석
- 입찰가 추천

### 1.2 핵심 가치

| 항목 | 개선 내용 |
|------|-----------|
| 분석 시간 단축 | 1시간 → 5분 |
| 권리 리스크 식별 | 자동화 |
| 입찰가 산정 | 자동 산출 |
| 투자 추천 | 개인화 |

### 1.3 타겟 사용자

- 단기 매매사업자
- 경매 투자자
- 부동산 중개/투자 전문가

## 2. 시스템 범위 (Phase 1 MVP)

### 2.1 대상 범위

| 구분 | 내용 |
|------|------|
| 물건 유형 | **아파트만** (오피스텔, 상가 등 제외) |
| 지역 | **서울 전체**, **경기도 용인시 수지구**, **경기도 성남시 분당구** |
| 입찰기일 필터 | **오늘(실행일) 이후 미래 기일인 물건만** 표시 |
| 데이터 소스 | 대법원 법원경매정보 (courtauction.go.kr) |

### 2.2 Phase 1 목표

> **"서울/용인수지/성남분당 지역 아파트 경매 물건을 실시간으로 수집하고, 미래 입찰 기일 기준으로 목록을 웹에서 즉시 조회한다"**

### 2.3 UI 방식

**Web (Streamlit)** 채택

| 구분 | Web (Streamlit) | Terminal |
|------|-----------------|----------|
| 테이블 정렬/필터 | 컬럼 클릭으로 재정렬, 필터 UI 제공 | 불편 |
| 법원 사이트 링크 | 클릭 가능 | 불가 |
| D-day/할인율 시각화 | 색상·배지 표시 가능 | 텍스트만 |
| 추후 기능 확장 | 용이 | 재구조화 필요 |

## 3. 기능 요구사항 (Phase 1)

### 3.1 경매 데이터 수집

**기능**
- 대법원 경매정보 사이트 크롤링 (Playwright 사용, JS 렌더링 필요)
- 지역 코드 기반 필터링 (서울 전체 / 용인수지구 / 성남분당구)
- 물건종류 = 아파트 필터
- 사건 단위 데이터 SQLite 저장

**수집 데이터**

| 필드 | 설명 |
|------|------|
| 사건번호 | 법원 사건 식별자 (법원 링크 생성에 사용) |
| 법원 | 관할 법원명 |
| 물건번호 | 물건 식별자 |
| 주소 | 소재지 (단지명 포함) |
| 물건종류 | 아파트 고정 |
| 감정가 | 감정평가 금액 (원) |
| 최저입찰가 | 입찰 최저 금액 (원) |
| 유찰횟수 | 누적 유찰 수 |
| 입찰기일 | 입찰 예정일 (YYYY-MM-DD) |
| 전용면적 | 전용면적 (㎡) |
| 층수 | 해당층 / 전체층 |
| 이미지 URL | 물건 사진 링크 (Phase 1 수집만, 미표시) |
| 상세 URL | 목록 크롤링 시 동적 추출한 상세 페이지 링크 (href/onclick) |
| 수집일시 | 크롤링 실행 시각 |

**요구사항**
- 크롤링 실패 시 최대 3회 retry
- 데이터 중복 방지 (사건번호 + 물건번호 기준 upsert)
- 변경 감지 및 업데이트 (유찰 발생, 최저가 변경)

---

### 3.2 경매 목록 표시

**표시 조건**
- 입찰기일이 오늘(실행일) 이후인 물건만 표시
- 기본 정렬: 입찰기일 오름차순
- 컬럼 클릭으로 재정렬 가능

**표시 컬럼 (9개)**

| # | 컬럼명 | 표시 형식 | 설명 |
|---|--------|-----------|------|
| 1 | 입찰기일 | `2026-04-10 (D-15)` | 날짜 + 남은 일수 배지 |
| 2 | 법원 / 사건번호 | 클릭 시 법원 사이트 이동 | 관할 법원 + 사건번호 링크 |
| 3 | 주소 | 텍스트 | 소재지 (단지명 포함) |
| 4 | 전용면적 | `84.97㎡ (25평)` | ㎡ + 평 환산 병기 |
| 5 | 층수 | `10 / 25층` | 해당층 / 전체층 |
| 6 | 감정가 | `5.2억` | 억원 단위 |
| 7 | 최저입찰가 | `3.6억` | 억원 단위 |
| 8 | 할인율 | `69%` | 최저가 / 감정가 × 100 (최저가율: 감정가 대비 최저가 비율) |
| 9 | 유찰횟수 | `2회` / `신건` | 0이면 "신건" 표시 |

**필터 UI (사이드바)**
- 지역 선택 (전체 / 서울 / 용인수지 / 성남분당)
- 입찰기일 범위 선택
- 할인율 하한 설정 (예: 70% 이상인 물건만 — 최저가가 감정가의 70% 이하인 물건)
- 유찰횟수 범위 설정

## 4. 데이터 모델

### 4.1 AuctionProperty (주 테이블)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `id` | INTEGER PK | 자동 증가 |
| `case_number` | TEXT | 사건번호 |
| `court` | TEXT | 관할 법원 |
| `property_number` | TEXT | 물건번호 |
| `address` | TEXT | 소재지 |
| `property_type` | TEXT | 물건종류 (아파트 고정) |
| `appraised_value` | INTEGER | 감정가 (원) |
| `min_bid_price` | INTEGER | 최저입찰가 (원) |
| `discount_rate` | REAL | 할인율 (계산값) |
| `failed_count` | INTEGER | 유찰횟수 |
| `bid_date` | DATE | 입찰기일 |
| `exclusive_area` | REAL | 전용면적 (㎡) |
| `current_floor` | INTEGER | 해당층 |
| `total_floor` | INTEGER | 전체층 |
| `image_url` | TEXT | 이미지 URL (Phase 1 수집만, 미표시) |
| `detail_url` | TEXT | 상세 페이지 URL (크롤링 시 동적 추출) |
| `region` | TEXT | 지역 구분 — enum: `서울`, `용인수지`, `성남분당` |
| `created_at` | DATETIME | 최초 수집일시 |
| `updated_at` | DATETIME | 최종 업데이트일시 |

## 5. 비기능 요구사항

### 5.1 성능

- 크롤링: 비동기 처리
- Streamlit 페이지 로드: 2초 이내

### 5.2 안정성

- 크롤링 실패 시 최대 3회 retry
- 데이터 정합성 검증 (감정가 > 0, 입찰기일 유효성)

### 5.3 확장성

- 모듈별 독립 구조 (crawler / db / app 분리)
- 지역 코드 설정 파일로 관리 (지역 추가 용이)

## 6. 시스템 아키텍처

### 6.1 기술 스택

| 구성 요소 | 기술 스택 | 비고 |
|-----------|-----------|------|
| UI | **Streamlit** | Python 기반 웹 UI |
| Crawling | **Playwright** | JS 렌더링 필요 |
| DB | **SQLite** | 단순, 무설치 |
| Backend | Python 스크립트 | FastAPI 불필요 |
| Storage | 로컬 파일 | S3 불필요 |

### 6.2 아키텍처 흐름

```
[Step 1: 크롤링 — CLI 별도 실행]
python -m crawler.court_auction

[대법원 경매정보 목록 페이지]
       │  Playwright (sync_playwright)
       │  - 목록 페이지 순회
       │  - 각 물건의 detail_url 동적 추출 (href/onclick)
       ▼
[SQLite DB]
  - AuctionProperty 테이블
  - Upsert (case_number + property_number 기준)

[Step 2: 조회 — Streamlit 실행]
streamlit run app.py

[Streamlit Web UI]  ← DB 읽기 전용 (크롤러 직접 호출 없음)
  - 입찰기일 >= today 필터
  - 9개 컬럼 테이블 (detail_url → 법원/사건번호 하이퍼링크)
  - 사이드바 필터 (지역/기일/할인율/유찰횟수)
  - 컬럼 클릭 정렬
```

### 6.3 디렉토리 구조

```
AuctionBrain/
├── crawler/
│   ├── __init__.py
│   └── court_auction.py     # Playwright 크롤러
├── db/
│   ├── __init__.py
│   ├── models.py             # SQLite 스키마
│   └── repository.py         # CRUD
├── app.py                    # Streamlit 앱 진입점
├── config.py                 # 지역 코드, 설정값
├── requirements.txt
└── data/
    └── auction.db            # SQLite 파일
```

## 8. 리스크 및 정책

### 8.1 리스크

| 리스크 | 대응 |
|--------|------|
| 대법원 사이트 크롤링 차단 | User-Agent 설정, 요청 간격 조절, retry |
| 데이터 불일치 (유찰 미반영 등) | 주기적 재수집, 변경 감지 로직 |
| 주소 파싱 오류 | 지역 코드 기반 필터로 보완 |

### 8.2 정책

- 크롤링은 대법원 공개 데이터만 수집
- 데이터 누락 시: 해당 물건 표시 보류
- LLM은 Phase 2 이후 도입 (Phase 1에서는 미사용)

## 9. 성공 기준

- 서울/용인수지/성남분당 지역 아파트 경매 목록 자동 수집
- 미래 입찰기일 기준 필터링 정확도 100%
- 할인율/D-day 계산 오류 없음
- 크롤링 후 Streamlit에서 30초 이내 결과 확인
